# ADR-003: Выбор базы данных — PostgreSQL

Статус: ✅ Принято

---

## Контекст

Системе необходимо хранить структурированные данные:
- Пользователи и профили
- Библиотека медитаций и категорий
- Сессии медитаций (история)
- Записи дневника настроения
- Прогресс пользователей

Данные имеют чёткую реляционную структуру.
Требуется поддержка транзакций и сложных запросов.

---

## Решение

**Backend:** PostgreSQL 15 + Spring Data JPA (Hibernate)  
**Android:** Room (SQLite) — локальный кэш

### Схема данных

categories ──< meditations  
│
users ─────────< meditation_sessions  
│
├──────────< mood_entries  
│
└────────── user_progress


---

## Обоснование

### PostgreSQL

| Критерий         | PostgreSQL                                     |
|------------------|------------------------------------------------|
| Тип              | Реляционная — соответствует структуре данных   |
| ACID             | Полная поддержка транзакций                    |
| JSON             | JSONB для гибких полей (теги медитаций)        |
| Интеграция       | Отличная поддержка Spring Data JPA             |
| Стоимость        | Бесплатная, open-source                        |

### Room (Android)

| Критерий         | Room                                           |
|------------------|------------------------------------------------|
| Официальность    | Google Jetpack — официальная библиотека        |
| Типобезопасность | Compile-time проверка SQL-запросов             |
| Coroutines       | Нативная поддержка Flow и suspend-функций      |
| Миграции         | Встроенные автоматические миграции схемы       |

### Конфигурация для тестов

```properties
# application-test.properties
spring.datasource.url=jdbc:h2:mem:testdb
spring.datasource.driver-class-name=org.h2.Driver
spring.jpa.database-platform=org.hibernate.dialect.H2Dialect
spring.jpa.hibernate.ddl-auto=create-drop
```

## Альтернативы

| СУБД            | Причина отказа                                 |
|-----------------|------------------------------------------------|
| MySQL           | Меньше возможностей, слабее поддержка JSON     |
| MongoDB         | NoSQL избыточен для реляционных данных         |
| H2              | In-memory — только для тестов                  |
| SQLite (сервер) | Не подходит для многопользовательского сервера |

## Последствия
### Положительные
- Единая реляционная модель на обеих платформах 
- Сильная типизация запросов 
- Офлайн-работа через Room-кэш
### Отрицательные
- Для локальной разработки нужен Docker или установленный PostgreSQL 
- Два маппинга: Domain ↔ Room Entity и Domain ↔ JPA Entity
