# ADR-004: Стратегия аутентификации - JWT

Статус: ✅ Принято

---

## Контекст

Мобильное приложение взаимодействует с REST API.
Необходимо:
- Идентифицировать пользователя при каждом запросе
- Stateless-архитектура сервера (без сессий)
- Безопасное хранение credentials на устройстве
- Работа без cookies (мобильный клиент)

---

## Решение

**JWT (JSON Web Tokens):** access token + refresh token  
**Хеширование паролей:** BCrypt (strength = 12)

### Параметры

| Параметр            | Значение                    |
|---------------------|-----------------------------|
| Access token TTL    | 15 минут                    |
| Refresh token TTL   | 7 дней                      |
| Алгоритм подписи    | HMAC-SHA256                 |
| Хранение на клиенте | DataStore Preferences (шифр)|
| BCrypt strength     | 12                          |

### Защита эндпоинтов
1. Публичные (без токена):  
POST /api/auth/register  
POST /api/auth/login  
POST /api/auth/refresh


2. Защищённые (ROLE_USER):  
GET  /api/meditations/**  
GET  /api/mood/**  
POST /api/mood  
GET  /api/users/me  

3. Только ROLE_ADMIN:  
POST   /api/meditations  
PUT    /api/meditations/{id}   
DELETE /api/meditations/{id}  


---

## Обоснование

| Критерий          | JWT                                          |
|-------------------|----------------------------------------------|
| Stateless         | Сервер не хранит сессии                      |
| Мобильный клиент  | Нет зависимости от cookies                   |
| Производительность| Проверка без обращения к БД                  |
| Стандарт          | RFC 7519 - широко поддерживается             |

---

## Альтернативы

| Механизм          | Причина отказа                                    |
|-------------------|---------------------------------------------------|
| Session + Cookie  | Не подходит для мобильных клиентов                |
| OAuth2            | Избыточно для учебного проекта                    |
| Basic Auth        | Небезопасно - пароль в каждом запросе             |
| API Key           | Нет механизма истечения срока                     |

---

## Последствия

### Положительные
- Stateless - нет хранилища сессий на сервере
- Access token истекает через 15 мин - минимальный риск
- BCrypt защищает пароли при утечке БД

### Отрицательные
- Access token нельзя инвалидировать досрочно (в v1.0)
- Секрет JWT в продакшене - обязательно через env variable
