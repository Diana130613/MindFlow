# ADR-001: Выбор архитектурного паттерна

Проект: MindFlow  
Дата: 29.05.2026  
Автор: Хатуаева Д.Т., ПИЖ-б-о-23-2  
Статус: ✅ Принято

---

## Контекст

Необходимо выбрать архитектурный паттерн для организации кода
на двух платформах одновременно:
- Android-клиент (Kotlin + Jetpack Compose)
- Backend-сервер (Java 17 + Spring Boot)

Требования к паттерну:
- Чёткое разделение ответственности между слоями
- Возможность тестировать каждый слой независимо
- Понятная структура для одного разработчика
- Демонстрация понимания архитектурных принципов (учебный проект)

---

## Решение

Использовать архитектуру **PCMEF**:

| Слой | Расшифровка  | Ответственность                        |
|------|-------------|----------------------------------------|
| P    | Presentation | Отображение UI, реакция на события    |
| C    | Control      | Обработка команд, валидация           |
| M    | Mediator     | Бизнес-логика, координация данных     |
| E    | Entity       | Бизнес-объекты, доменные правила      |
| F    | Foundation   | Доступ к данным (БД, сеть)            |

### Правила зависимостей
✅ РАЗРЕШЕНО:        ❌ ЗАПРЕЩЕНО:
P → C               P → M (минуя C)
C → M               P → F (минуя C и M)
M → E               C → F (минуя M)
M → F               F → M (обратная зависимость)
E → (ничего)        E → F (Entity не знает о БД)



### Маппинг на платформы

| Слой PCMEF     | Android                         | Backend                  |
|----------------|---------------------------------|--------------------------|
| Presentation P | Composable-функции, Activity    | —                        |
| Control C      | ViewModel + StateFlow           | REST Controllers         |
| Mediator M     | RepositoryImpl (domain)         | ServiceImpl              |
| Entity E       | Data классы (domain)            | JPA Entity               |
| Foundation F   | Room DAO + Retrofit API Service | JPA Repository (Spring)  |

---

## Обоснование

1. **Явные границы** — каждый класс однозначно относится к одному слою
2. **Однонаправленные зависимости** — изменения в F не затрагивают P
3. **Тестируемость** — M тестируется с моком F, C тестируется с моком M
4. **Единый паттерн** для двух платформ упрощает понимание проекта
5. **Масштабируемость** — новые функции добавляются без изменения
   существующих слоёв

---

## Альтернативы, которые рассматривались

| Паттерн           | Причина отказа                                          |
|-------------------|---------------------------------------------------------|
| MVC               | Controller разрастается, слабое разделение             |
| MVP               | Устарел для Android, сложное тестирование Presenter    |
| Clean Architecture| Избыточен: слишком много слоёв для учебного проекта    |
| MVI               | Сложная реализация, избыточен для данного масштаба     |
| MVVM (только)     | Не определяет структуру M и F слоёв                   |

---

## Последствия

### Положительные
- Чёткая структура пакетов в каждом модуле
- Простое ревью кода — сразу понятно, где искать логику
- Изолированное тестирование каждого слоя

### Отрицательные / Ограничения
- Требует дисциплины — нельзя нарушать направление зависимостей
- Небольшой overhead: для каждой фичи нужны классы во всех слоях
- Начинающим разработчикам может потребоваться время на освоение
