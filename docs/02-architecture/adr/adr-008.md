# ADR-008: Асинхронность — Coroutines + Flow

Статус: ✅ Принято

---

## Контекст

Android запрещает сетевые и IO-операции в главном потоке.
Необходим механизм асинхронного программирования для:
- Запросов к Room (IO поток)
- Сетевых запросов через Retrofit (IO поток)
- Реактивного обновления UI при изменении данных
- Фоновой синхронизации (WorkManager)

---

## Решение

Использовать **Kotlin Coroutines** + **Flow** для реактивных
потоков данных.

### Паттерн использования по слоям

```kotlin
// F-слой (Foundation) — Room возвращает Flow
@Dao
interface MoodEntryDao {
    @Query("SELECT * FROM mood_entries WHERE user_id = :userId")
    fun getByUser(userId: Long): Flow<List<MoodEntryDbEntity>>

    @Insert(onConflict = OnConflictStrategy.REPLACE)
    suspend fun insert(entry: MoodEntryDbEntity)
}

// M-слой (Mediator) — Repository трансформирует Flow
class MoodRepositoryImpl(
    private val dao: MoodEntryDao,
    private val api: MindFlowApiService
) : IMoodRepository {

    override fun getMoodHistory(days: Int): Flow<List<MoodEntry>> =
        dao.getByUser(userId)
            .map { list -> list.map { it.toDomain() } }
            .flowOn(Dispatchers.IO)

    override suspend fun saveMoodEntry(entry: MoodEntry) {
        val dbEntity = entry.toDbEntity(syncPending = true)
        dao.insert(dbEntity)
        try {
            api.saveMoodEntry(entry.toRequest())
            dao.markSynced(entry.id)
        } catch (e: IOException) {
            // Сохранено локально — синхронизируем позже
        }
    }
}

// C-слой (Control) — ViewModel собирает Flow в StateFlow
class MoodViewModel(
    private val moodRepository: IMoodRepository
) : ViewModel() {

    val moodHistory: StateFlow<List<MoodEntry>> =
        moodRepository.getMoodHistory(30)
            .stateIn(
                scope = viewModelScope,
                started = SharingStarted.WhileSubscribed(5000),
                initialValue = emptyList()
            )

    fun saveMood(score: Int, note: String) {
        viewModelScope.launch {
            moodRepository.saveMoodEntry(
                MoodEntry(score = score, note = note)
            )
        }
    }
}

// P-слой (Presentation) — Compose подписывается на StateFlow
@Composable
fun MoodDiaryScreen(viewModel: MoodViewModel = hiltViewModel()) {
    val moodHistory by viewModel.moodHistory.collectAsStateWithLifecycle()

    MoodDiaryContent(
        entries = moodHistory,
        onSave = { score, note -> viewModel.saveMood(score, note) }
    )
}
```

---

## Диспетчеры

| Диспетчер             | Использование                         |
|-----------------------|---------------------------------------|
| `Dispatchers.Main`    | Обновление UI, ViewModel              |
| `Dispatchers.IO`      | Room, Retrofit, файловые операции     |
| `Dispatchers.Default` | CPU-интенсивные операции (сортировка) |

---

## Обоснование

| Критерий            | Coroutines + Flow                             |
|---------------------|-----------------------------------------------|
| Нативность          | Встроен в Kotlin — не внешняя зависимость     |
| Структурированность | Структурированный параллелизм — нет утечек    |
| Реактивность        | Flow идеально подходит для реактивного UI     |
| Интеграция          | Нативная поддержка Room, Retrofit, ViewModel  |
| Читаемость          | Код выглядит синхронным, но выполняется async |

---

## Альтернативы
| Подход           | Причина отказа                                     |
|------------------|----------------------------------------------------|
| RxJava           | Сложный API, избыточен при наличии Flow            |
| AsyncTask        | Устарел — удалён в API 30                          |
| LiveData         | Менее мощный чем Flow, только для UI-слоя          |
| Thread + Handler | Низкоуровневый, много шаблонного кода, риск утечек |

---

## Последствия
### Положительные
- Код читается как синхронный, но выполняется асинхронно
- viewModelScope автоматически отменяет задачи при уничтожении ViewModel
- Flow из Room автоматически обновляет UI при изменении данных в БД
## Отрицательные
- Требует понимания концепции structured concurrency
- Неправильный выбор диспетчера может вызвать ANR
